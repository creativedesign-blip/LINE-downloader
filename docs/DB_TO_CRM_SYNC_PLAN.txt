 0. TL;DR (the decided design in one paragraph)
================================================================================
Push ONE-WAY (local -> CRM), near-real-time. Desktop is behind NAT (outbound
only), so the desktop PUSHES out over an EPHEMERAL SSH TUNNEL (built-in ssh.exe)
to the GCE VM, where MySQL listens only on 127.0.0.1. Every sync run uses ONE
unified mechanism for BOTH source DBs: "snapshot reconciliation" — read all
local rows, hash each, diff against a local SHADOW of what was last pushed,
UPSERT changed rows and DELETE rows that disappeared, then update the shadow.
MySQL is a pure write target (no read-back). Empty shadow = first run = full
backfill automatically. Implemented as a new self-contained package tools/sync/
that does NOT modify the app's existing DB schemas. Triggered after each
RPA/upload run + a periodic safety-net thread + a CLI entrypoint.

 3. OPTIONS CONSIDERED AT EACH DECISION POINT (with trade-offs + what was chosen)

3.1 TRANSPORT / DIRECTION
  (A) Periodic ETL push (dump -> GCS -> BigQuery/Cloud SQL). Simple, decoupled,
      but not near-real-time and lands in a separate store.
  (B) Incremental push to a CRM ingest API. Near-real-time, needs an API on GCP.
  (C) SQLite replica to cloud (Litestream -> GCS). Good as backup; CRM can't
      easily query a SQLite file -> rejected.
  (D) Reverse-expose the desktop DB (tunnel/VPN so CRM pulls). Fragile, security
      risk on a home box -> rejected.
  CHOSEN DIRECTION: PUSH outbound from desktop (variant of B). Fits existing
  scheduler; no NAT/inbound exposure.

3.2 MYSQL TYPE (informs the connection method)
  - GCE VM self-hosted  <-- ACTUAL (user confirmed "該機器 mysql")
  - Cloud SQL for MySQL  (not used; would have used cloud-sql-python-connector)

3.3 CONNECTION METHOD (desktop -> VM MySQL)
  (1) Native ssh.exe SSH tunnel  <-- CHOSEN. Already installed
      (C:\Windows\System32\OpenSSH\ssh.exe and Git's ssh). No GCP deploy. Auth by
      SSH key. VM already has sshd:22.
  (2) gcloud IAP TCP forwarding. More secure (no public 22, Google IAM auth) but
      gcloud is NOT installed on the box; requires SDK + service account. Noted as
      a future hardening upgrade — only the "open tunnel" step changes.
  (3) Ingest API in front (Cloud Run/Function). Most decoupled; extra service to
      build/maintain + VPC wiring to reach the VM's MySQL -> overkill for scale.
  RULE: NEVER expose MySQL 3306 to the public internet. MySQL stays bound to
  127.0.0.1 on the VM; reach it only through the tunnel.

3.4 SYNC MECHANISM (how to know what changed)
  (i)   Watermark only (updated_at/indexed_at > last). Simplest but MISSES hard
        deletes; manual_tags has no updated_at.
  (ii)  Outbox table + SQLite triggers (AFTER I/U/D) on catalog tables; watermark
        + key-set reconciliation on the rebuilt index DB. Event-driven & precise,
        but: must edit the app's init_db(), maintain triggers, and handle the
        FK-cascade/foreign_keys=OFF caveat. Considered, then REJECTED as
        over-engineered for the data volume.
  (iii) UNIFIED SNAPSHOT RECONCILIATION  <-- CHOSEN. One mechanism for both DBs;
        no triggers; does not touch app schema; handles deletes; backfill == empty
        shadow. Cost: reads full tables each run to hash (fine at current scale,
        thousands of rows, sub-second).
  DECISION FACTS that make (iii) safe re: the index DB rebuild — verified in code:
    * tools/indexing/index_db.py TravelIndex.__init__: on PRAGMA user_version
      mismatch with migrate=True it DROPs+recreates the 3 itinerary tables.
    * tools/indexing/reindex.py: normal path = per-sidecar index.upsert(); the
      `--rebuild` path does index.clear() + re-insert INSIDE a single
      index.transaction() (commit once at end).
    * Because clear+reinsert is ONE transaction and the DBs are WAL, a separate
      read connection (the sync) only ever sees a COMPLETE committed snapshot
      (pre- or post-rebuild), NEVER the empty mid-state. So reconciliation will
      not "delete everything" mid-rebuild. A 0-row delete GUARD is still added as
      insurance against a future non-atomic change.

3.5 SHADOW / STATE STORAGE
  - Earlier (trigger design) idea: keep state in catalog.db. With reconciliation,
    chosen approach is a DEDICATED, fully-decoupled metadata DB:
      logs/openclaw/sync_meta.db   (holds the shadow + run status)
    Source DBs are opened SELECT-ONLY; their schema is never modified.

================================================================================
 4. FINAL ARCHITECTURE
================================================================================
  SQLite (source of truth, read-only)                 GCE VM
    travel_index.db  ----,                              sshd:22
    upload_catalog.db ---+--> tools/sync/ run_once()      |
                              |  diff vs shadow            | ssh.exe -N -L
   sync_meta.db (shadow,     |  (sync_meta.db)            | 13306:127.0.0.1:3306
   status) <-----------------+  open ephemeral tunnel ----v
                              |  PyMySQL upsert/delete --> MySQL (CRM mirror, 127.0.0.1)
                              |  on commit: update shadow + status
   TRIGGERS: (a) after each RPA run  (b) after each upload run
             (c) periodic safety-net thread  (d) CLI / Task Scheduler

Per-table reconciliation algorithm (the heart of it):
  1. SELECT mapped columns for all rows of the source table (batched ~500).
  2. row_hash = sha256(canonical encoding of mapped column values) per row.
  3. Load shadow {pk -> row_hash} for this table from sync_meta.db.
  4. UPSERTS = rows whose pk is new OR whose hash changed.
     DELETES = pks present in shadow but absent locally.
     (GUARD: if local row count == 0 but shadow non-empty, SKIP deletes + warn.)
  5. writer.upsert(upserts); writer.delete(deletes)  (per-batch transactions).
  6. On MySQL commit success, write the new full snapshot to shadow.
Principles: NON-FATAL (never breaks an RPA/upload run), IDEMPOTENT (safe to
re-run), SERIALIZED (a non-blocking lock prevents post-run + timer overlap).

 11. RUNTIME FLOW of one run_once()
================================================================================
  1. is_enabled()? no -> return.
  2. Acquire non-blocking lock; busy -> log "skipped" + return.
  3. cfg = config.load(); missing env -> write_status(error) + return.
  4. meta.ensure_tables().
  5. retry loop (2-3, backoff):
       a. open SSHTunnel (ephemeral, hardened args).
       b. writer.connect() to 127.0.0.1:LOCAL_PORT.
       c. writer.preflight(TABLES); mismatch -> write_status(error) + ABORT (no retry).
       d. for each mapping in mappings.TABLES:
            open source DB SELECT-only -> reconcile_table(...) -> TableSummary.
       e. writer.close(); tunnel teardown (guaranteed).
  6. success -> meta.write_status(success, counts);
     exception -> meta.write_status(error); NEVER raised to caller.

 15. RISKS / EDGE CASES
================================================================================
- Schema drift: mappings.py is the single source of truth; preflight fails loudly.
- Partial failure: per-batch transactions; shadow advanced only after commit ->
  safe auto-retry next run.
- Shadow vs MySQL drift: assumes MySQL not edited externally (one-way contract).
  Manual CRM edits to synced rows get overwritten/reconciled. A periodic
  read-back verify pass can be added later if needed.
- SSH: passphrase-less key required; host key pinned; BatchMode avoids hangs.
- Secrets: env-only, never logged; MySQL user scoped (not root).
- Reconcile cost: reads all rows each run to hash; thousands -> sub-second. If
  tables grow large, add an updated_at/indexed_at fast-path to skip unchanged rows.
- Outbox growth: N/A (no outbox in the chosen design).
- Timestamp compare: indexed_at/updated_at are ISO8601Z TEXT; lexicographic order
  is valid only if the format is consistent. (Reconciliation uses hashes, not
  timestamps, so this only matters if a fast-path is added later.)
