 1. THE BIG PICTURE (where the chatbot fits)
================================================================================
Three segments, three owners:

  [Desktop SQLite]  --(1) auto-sync-->  [Your MySQL on GCP]  --(2) query-->  [Chatbot answers]
   (travel data source)                  (a near-live mirror)                 (GPT + your code)
        DB/data team                      filled by DB team                   YOUR TEAM
                                          read by YOUR team

- Segment (1): DB/data team copies the travel data into YOUR MySQL every few
  minutes. NOT your concern beyond agreeing on columns.
- Segment (2): YOUR job. The chatbot reads from MySQL and answers customers.
  THIS DOCUMENT IS ABOUT SEGMENT (2).

KEY FACTS you must internalize:
- The data ALREADY lands in YOUR OWN MySQL. Reading it is a normal database
  query (SELECT) — NOT calling an external/3rd-party API. There is NO API on the
  DB team's side for you to call. The MySQL table IS the handoff point.
- The data is READ-ONLY for you. Only the sync writes to these tables. If your
  bot writes/edits them, the next sync overwrites it. Store chatbot-owned data
  (chat logs, sessions) in SEPARATE tables you create.
- Freshness: a change on the desktop shows up in your MySQL within a few minutes
  (near-real-time, not millisecond-live). Fine for a travel chatbot.

 2. DO I NEED TO BUILD AN "API"?  (the recurring question, answered once)
================================================================================
The word "API" means two different things; do not mix them up:

  (A) A WEB API = an HTTP endpoint/URL you call, e.g. GET /search?country=日本.
  (B) A DATABASE DRIVER = a code library your program uses to talk to MySQL,
      e.g. pymysql (Python) / mysql2 (Node). People do NOT usually call this
      "an API".

The step "go to MySQL and run the query" is (B): your backend program connects
to MySQL and runs SQL directly. It is NOT a web API. There is nothing to "call"
on anyone else's server.

When (and only when) you ALSO build a web API (A):
  - Your bot logic and your DB access run as SEPARATE services/servers, OR
  - Multiple apps must share the same query logic, OR
  - You want an extra isolation/security boundary.
If your chatbot is ONE backend program, you do NOT need a web API — connect to
MySQL directly. Any web API you build would be YOUR OWN internal wrapper around
YOUR OWN MySQL, still not a call to the DB team.

ALSO CRITICAL: GPT does NOT touch the database and does NOT call any API itself.
In "function calling" GPT only OUTPUTS a JSON like {country:"日本"}; YOUR code
reads that and runs the actual query. "GPT queries the DB" is a metaphor — your
code always does the real query.

 3. RECOMMENDED ARCHITECTURE : GPT FUNCTION CALLING + DIRECT SQL
================================================================================
Flow of a single customer message:

  customer message
     -> (your code) send message history + a "search tool" definition to GPT
     -> GPT decides: is this a travel-search intent?
          - NO  (small talk)  -> GPT just replies normally. No DB hit.
          - YES               -> GPT returns a tool call with extracted params,
                                  e.g. {country:"日本", keywords:["大阪","賞櫻"]}
     -> (your code) run a parameterized SQL SELECT on MySQL with those params
     -> (your code) send the rows back to GPT as a "tool result"
     -> GPT writes the final natural-language answer USING ONLY those rows
     -> reply to customer

Why this beats naive keyword matching:
  - GPT decides WHEN to query (won't fire on "I went to Japan last year, it was
    fun"); won't miss rephrasings/synonyms/typos.
  - GPT extracts structured params for you (country, month, budget) AND free-text
    keywords for things that have no dedicated column.
  - You write LESS glue code than hand-rolled keyword rules.

Alternatives considered (and why not, for your case):
  - Pure keyword rules -> SQL : brittle, over/under-triggers. Difficulty: low,
    quality: poor.
  - Text-to-SQL (GPT writes raw SQL) : flexible but risky (can generate wrong or
    unsafe SQL, injection/permission concerns). Difficulty: high. Avoid for now.
  - RAG / vector semantic search : great for "find similar / by vibe", but your
    queries have hard filters (country, month, price) better served by SQL.
    Overkill now; can add later for fuzzy discovery.

4.1 *** SPECIAL COLUMN FORMAT (must read) ***
Columns ending in "_csv" can hold MULTIPLE values and are stored COMMA-WRAPPED:
    country_csv = ",日本,韓國,"      months_csv = ",6,7,8,"
To test membership, DO NOT use "= '日本'". Use a wrapped LIKE:
    country_csv LIKE '%,日本,%'      months_csv LIKE '%,7,%'
The leading/trailing commas make it an EXACT token match (so "日本" won't wrongly
match "日本海"). Always wrap your search value as '%,VALUE,%' for these columns.

================================================================================
 5. KEYWORD -> COLUMN MAPPING (the real-world nuance)
================================================================================
Customer words map to the DB in three different ways. Plan for all three.

  Customer says   ->  How to match
  --------------------------------------------------------------------------
  日本 (country)  ->  has a column: country_csv LIKE '%,日本,%'   (precise)
  7月 (month)     ->  has a column: months_csv LIKE '%,7,%'
  五萬以內(budget)->  has a column: price_from <= 50000
  5天(duration)   ->  has a column: duration_days <= 5  (or =)
  大阪 (city)     ->  NO city column. Search free text: group_name / region_csv /
                      features_csv / (title, raw_text if synced) LIKE '%大阪%'
  賞櫻 (theme)    ->  usually in features_csv (or title/raw_text). LIKE '%賞櫻%'
  優惠 (promo)    ->  NO field today. See section 6.

RULE OF THUMB:
  - Words that have a column  -> add a precise WHERE condition.
  - Words that don't         -> push into a "keywords" list and LIKE-search across
                                the text columns.
GPT (via function calling) should output BOTH: structured params AND a keywords[]
array. Your query builder then handles each appropriately (see section 8).

================================================================================
 6. THE "優惠 / PROMOTION" PROBLEM (decide with the DB team)
================================================================================
There is currently NO discount/promotion field in the data. Options:
  (a) Treat "優惠" as "show the cheapest" -> ORDER BY price_from ASC, return top N.
  (b) Ask the DB team to ADD a promo field (e.g. is_promo / promo_price) to the
      sync, then filter on it precisely. (Cleanest long-term.)
  (c) Ignore the word as a filter.
Pick one. Until a real field exists, (a) is the pragmatic default.

8.1 DB CONNECTION (read-only user)
--------------------------------------------------------------------------------
  import pymysql
  conn = pymysql.connect(
      host="<MYSQL_HOST>", port=3306,
      user="<READONLY_USER>", password="<PASSWORD>",
      database="<DB_NAME>", charset="utf8mb4",
      cursorclass=pymysql.cursors.DictCursor,  # rows come back as dicts
      autocommit=True,
  )
  # In production, use a CONNECTION POOL (e.g. DBUtils PooledDB) and read creds
  # from environment variables, never hard-coded. (Node: mysql2 createPool.)

8.2 THE QUERY BUILDER (structured filters + free-text keywords)
--------------------------------------------------------------------------------
  def search_itineraries(country=None, month=None, price_max=None,
                         duration_max=None, keywords=None, promo=False, limit=10):
      sql = """SELECT sidecar_path, group_name, country_csv, region_csv,
                      months_csv, features_csv, price_from, duration_days
               FROM itineraries WHERE 1=1"""
      params = []
      if country:                                   # column exists -> precise
          sql += " AND country_csv LIKE %s"; params.append(f"%,{country},%")
      if month:
          sql += " AND months_csv LIKE %s";  params.append(f"%,{int(month)},%")
      if price_max:
          sql += " AND price_from <= %s";    params.append(int(price_max))
      if duration_max:
          sql += " AND duration_days <= %s"; params.append(int(duration_max))
      for kw in (keywords or []):                   # no column -> scan text cols
          sql += (" AND (group_name LIKE %s OR features_csv LIKE %s"
                  " OR region_csv LIKE %s)")
          like = f"%{kw}%"; params += [like, like, like]
      # "promo" has no real field yet -> approximate as cheapest-first (option a)
      sql += " ORDER BY price_from ASC" if promo else " ORDER BY price_from ASC"
      sql += " LIMIT %s"; params.append(int(limit))
      with conn.cursor() as cur:
          cur.execute(sql, params)                  # ALWAYS parameterized (%s)
          return cur.fetchall()

  *** SECURITY: never build SQL by string-concatenating user input. Always pass
      values through %s placeholders + params (prevents SQL injection). ***

  KEYWORD SEMANTICS NOTE: the loop ANDs every keyword (all must match) -> precise
  but can return 0 rows for many keywords. If you get 0 results, consider a second
  pass that relaxes (OR the keywords, or drop the least important) and tell the
  customer you broadened the search.

8.3 DEFINE THE TOOL FOR GPT (function calling)
--------------------------------------------------------------------------------
  tools = [{
    "type": "function",
    "function": {
      "name": "search_itineraries",
      "description": ("查旅遊行程資料庫。當客人想找行程(提到國家、城市、主題、"
                      "月份、天數、預算、優惠等)時呼叫。閒聊時不要呼叫。"),
      "parameters": {
        "type": "object",
        "properties": {
          "country":      {"type": "string",  "description": "國家,如 日本"},
          "month":        {"type": "integer", "description": "出發月份 1-12"},
          "price_max":    {"type": "integer", "description": "每人預算上限(台幣)"},
          "duration_max": {"type": "integer", "description": "天數上限"},
          "keywords":     {"type": "array", "items": {"type": "string"},
                           "description": "無專屬欄位的自由詞,如 大阪、賞櫻、溫泉"},
          "promo":        {"type": "boolean", "description": "客人是否想要優惠/便宜"}
        }
      }
    }
  }]

8.4 THE CONVERSATION LOOP (extract -> query -> answer)
--------------------------------------------------------------------------------
  import json
  from openai import OpenAI
  client = OpenAI()  # reads OPENAI_API_KEY from env

  SYSTEM = ("你是旅遊客服助理。需要找行程時呼叫 search_itineraries。"
            "回答只能根據工具回傳的資料;沒有資料就誠實說沒有,不要自己編造"
            "行程或價格。")

  def handle(messages):  # messages = full chat history (list of role/content)
      messages = [{"role": "system", "content": SYSTEM}] + messages
      r1 = client.chat.completions.create(
              model="gpt-4o", messages=messages, tools=tools)
      m = r1.choices[0].message
      if not m.tool_calls:
          return m.content                       # small talk -> direct reply
      # GPT asked to search:
      call = m.tool_calls[0]
      args = json.loads(call.function.arguments) # the params GPT extracted
      rows = search_itineraries(**args)          # YOUR code hits MySQL
      messages.append(m)                          # record GPT's tool call
      messages.append({"role": "tool", "tool_call_id": call.id,
                       "content": json.dumps(rows, ensure_ascii=False)})
      r2 = client.chat.completions.create(model="gpt-4o", messages=messages)
      return r2.choices[0].message.content       # final answer to customer

  Notes:
   - Pass the WHOLE message history so multi-turn context works
     ("那再便宜一點的呢?" relies on the previous turn).
   - GPT may request MULTIPLE tool calls; loop over m.tool_calls if you support
     more than one tool.
   - Keep the returned row set small (LIMIT 10) so you don't blow the token budget
     when feeding rows back to GPT.

  SELECT ... FROM itineraries
  WHERE country_csv LIKE '%,日本,%'

    AND (group_name LIKE '%大阪%' OR features_csv LIKE '%大阪%' OR region_csv LIKE '%大阪%')
    AND (group_name LIKE '%賞櫻%' OR features_csv LIKE '%賞櫻%' OR region_csv LIKE '%賞櫻%')
  ORDER BY price_from ASC LIMIT 10;
Your code runs it, gets rows, hands them back to GPT, GPT writes the reply using
only those rows.

================================================================================
 10. EDGE CASES & UX
================================================================================
- ZERO results: don't dead-end. Re-query with relaxed filters (drop a keyword,
  widen month) and tell the customer you broadened it; or ask a clarifying
  question ("要不要看其他月份?").
- HALLUCINATION GUARD: the system prompt must force "answer only from tool data".
  Optionally double-check that any price/itinerary the bot mentions exists in the
  returned rows before sending.
- PRESENTATION: decide how many to show (e.g. top 3-5), what fields (name, price,
  duration), and whether to link an image (image_path/branded_path -> you must
  host/serve those; raw filesystem paths are not web URLs).
- DEDUP: itineraries vs plans vs departures can describe the same trip at
  different granularity. Decide which table is your primary search target
  (usually itineraries for "find a trip", departures for "find a date").

 11. SECURITY & PERFORMANCE CHECKLIST
================================================================================
- [ ] Read-only MySQL user (SELECT only). Never root.
- [ ] Credentials from environment variables / secret manager, never in code/git.
- [ ] ALL queries parameterized (%s + params). No string concatenation of input.
- [ ] Do NOT expose MySQL 3306 to the public internet (use private network/tunnel).
- [ ] Connection pooling for concurrency (don't open a new connection per message).
- [ ] LIMIT every query; never SELECT * without a cap.
- [ ] The DB team maintains indexes on country_csv/months_csv/price_from/etc.
      (they exist in the source); confirm they're present on the MySQL side too
      so filtered queries stay fast.
- [ ] Cache hot/repeated queries if traffic is high (optional).

================================================================================
 15. PLAIN-LANGUAGE GLOSSARY
================================================================================
- MySQL          : the database that holds the travel data you query. It's YOURS;
                   the sync just fills it.
- SQL / SELECT   : the query language; SELECT = "read/fetch rows".
- Database driver: a code library (pymysql / mysql2) your program uses to talk to
                   MySQL. Not a "web API".
- Web API        : an HTTP endpoint (a URL) you call over the network. You do NOT
                   need one to read your own MySQL.
- Function calling / tool use : a GPT feature where GPT, instead of answering
                   directly, outputs "I want to call search_itineraries with
                   these arguments". YOUR code runs the real query. GPT never
                   touches the DB.
- Parameterized query : passing user values via %s placeholders instead of
                   gluing them into the SQL string -> prevents SQL injection.
- SQL injection  : an attack where malicious input changes your query; prevented
                   by parameterized queries.
- Hallucination  : when the LLM makes up facts (e.g. a fake itinerary/price);
                   prevented by "answer only from the returned data".
- _csv column    : a column storing multiple comma-wrapped values (",日本,韓國,");
                   match with LIKE '%,VALUE,%'.
- Near-real-time : changes appear within a few minutes, not instantly.
- RAG / vector search : semantic "find similar" search; not needed for your
                   structured filters, optional future enhancement.
